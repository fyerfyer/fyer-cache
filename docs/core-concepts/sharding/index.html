Sharding Principles

分片(Sharding)是FyerCache的核心设计之一，通过将数据和负载分布到多个独立单元上，实现高性能、高并发和可扩展性。FyerCache实现了两个层次的分片机制：内存分片和节点分片，分别解决单机并发和分布式扩展问题。

内存分片 - ShardedMap​

内存分片是FyerCache的基础分片机制，解决单机多核并发读写问题。

核心设计​

ShardedMap 将单一map拆分为多个独立分片，每个分片有自己的读写锁：

// ShardedMap 分片映射结构
type ShardedMap struct {
    shards    []*Shard     // 分片数组
    count     int          // 分片数量
    mask      uint32       // 位掩码，用于快速计算分片索引
    itemCount int64        // 总项目数量
}

// Shard 单个分片
type Shard struct {
    items map[string]*cacheItem // 实际存储的键值对
    mu    sync.RWMutex          // 分片级锁
}

分片算法​

FyerCache使用FNV哈希算法和位运算快速确定键所属分片：

// getShard 根据键获取对应的分片
func (sm *ShardedMap) getShard(key string) *Shard {
    hasher := fnv.New32()
    hasher.Write([]byte(key))
    hash := hasher.Sum32()

    // 使用位掩码快速计算分片索引
    index := hash & sm.mask
    return sm.shards[index]
}

位掩码计算相比取模运算更高效，要求分片数量是2的幂。

分片数量选择​

FyerCache的默认分片数量是32，用户可以根据硬件特性和性能需求进行调整：

// 通过选项设置分片数量
cache := cache.NewMemoryCache(
    cache.WithShardCount(64), // 使用64个分片
)

分片数量通常设置为CPU核心数的倍数，以获得最佳性能。如果设置值不是2的幂，系统会向上取整到最近的2的幂。

分片操作​

FyerCache对分片的核心操作包括：

  • Store - 将键值对存储到对应分片
  • Load - 从对应分片加载键值
  • Delete - 从对应分片删除键值
  • Range - 遍历所有分片中的键值对
  • RangeShard - 仅遍历特定分片中的键值对

分片锁只在操作特定分片时获取，大大减少了锁竞争：

// Store 存储键值对
func (sm *ShardedMap) Store(key string, value *cacheItem) {
    shard := sm.getShard(key)
    shard.mu.Lock()
    defer shard.mu.Unlock()

    // 存储操作...
}

// Load 加载指定键的值
func (sm *ShardedMap) Load(key string) (*cacheItem, bool) {
    shard := sm.getShard(key)
    shard.mu.RLock()
    defer shard.mu.RUnlock()

    // 加载操作...
}

节点分片 - NodeShardedCache​

节点分片构建在内存分片之上，支持跨多个节点分布数据，是FyerCache分布式能力的基础。

核心设计​

NodeShardedCache 使用一致性哈希将数据分布到多个缓存节点：

// NodeShardedCache 基于节点的分片缓存实现
type NodeShardedCache struct {
    // 一致性哈希环
    ring Ring

    // 节点ID到缓存的映射
    nodes map[string]Cache

    // 保护节点映射的并发访问
    mu sync.RWMutex

    // 备份因子，每个键存储在多少个节点上（用于容错）
    replicaFactor int
}

节点管理​

NodeShardedCache支持动态添加和移除节点：

// 创建节点分片缓存
shardedCache := cache.NewNodeShardedCache()

// 添加三个节点，每个有不同权重
shardedCache.AddNode("node1", cache1, 1)  // 权重1
shardedCache.AddNode("node2", cache2, 2)  // 权重2
shardedCache.AddNode("node3", cache3, 3)  // 权重3

// 移除节点
shardedCache.RemoveNode("node2")

节点权重影响数据分布，权重越高，分配给该节点的数据比例越大。

一致性哈希实现​

一致性哈希是节点分片的核心算法，确保数据均匀分布并最小化节点变更时的数据迁移：

// ConsistentHash 一致性哈希环实现
type ConsistentHash struct {
    // 哈希函数
    hashFunc HashFunc

    // 每个物理节点对应的虚拟节点数量
    replicas int

    // 已排序的哈希值
    sortedHashes []uint32

    // 哈希值到节点的映射
    hashMap map[uint32]string

    // 节点权重
    weights map[string]int

    // 节点到其哈希值的映射，用于高效删除
    nodeHashes map[string][]uint32

    // 保护环的并发访问
    mu sync.RWMutex
}

虚拟节点机制​

为提高数据均匀性，FyerCache实现了虚拟节点机制，默认每个物理节点对应100个虚拟节点：

// 创建节点分片缓存时设置虚拟节点数量
shardedCache := cache.NewNodeShardedCache(
    cache.WithNodeHashReplicas(200), // 每个物理节点200个虚拟节点
)

数据路由​

当需要存储或检索数据时，一致性哈希确定键所属的节点：

// Get 从正确的节点获取缓存项
func (nsc *NodeShardedCache) Get(ctx context.Context, key string) (any, error) {
    // 检查是否有节点
    if nsc.GetNodeCount() == 0 {
        return nil, ferr.ErrNoNodesAvailable
    }

    // 获取包含该键的所有节点（主节点和备份节点）
    nodeIDs := nsc.ring.GetN(key, nsc.replicaFactor)
    // ...获取数据逻辑...
}

副本机制​

为提高可用性，FyerCache支持将数据复制到多个节点：

// 创建带有3个副本的节点分片缓存
shardedCache := cache.NewNodeShardedCache(
    cache.WithNodeReplicaFactor(3), // 每个键存储在3个节点上
)

当某个节点失败时，系统可以从备份节点检索数据：

// 尝试从所有可能的节点获取数据
func (nsc *NodeShardedCache) Get(ctx context.Context, key string) (any, error) {
    // ...前面的代码...
    
    // 首先尝试主节点
    if val, err := nodeCache.Get(ctx, key); err == nil {
        return val, nil
    }
    
    // 如果主节点失败，尝试从备份节点获取
    for i := 1; i < len(nodes) && i < len(nodeIDs); i++ {
        if val, err := nodes[i].Get(ctx, key); err == nil {
            return val, nil
        }
    }
    
    return nil, ferr.ErrKeyNotFound
}

节点变更影响​

一致性哈希的关键优势是节点变更时最小化数据迁移：

  • 当节点加入或离开时，只有环上相邻节点负责的一小部分键需要重新分配
  • 理论上，添加或删除N节点集群中的1个节点，只有1/N的数据需要重新分配

分布式集群中的分片​

在完整的分布式系统中，FyerCache结合了节点分片和集群管理功能，提供更强大的分布式缓存功能。

ShardedClusterCache​

ShardedClusterCache 整合了一致性哈希分片与集群管理：

// ShardedClusterCache 整合一致性哈希分片与集群功能的缓存实现
type ShardedClusterCache struct {
    // 本地缓存实例
    localCache cache.Cache

    // 分片缓存实例，用于路由请求
    shardedCache *cache.NodeShardedCache

    // 集群节点
    node *cluster.Node

    // 节点分发器
    nodeDistributor *NodeDistributor
    
    // ...其他字段...
}

该实现提供以下功能：

  1. 动态集群成员管理 - 节点可以动态加入和离开集群
  2. 自动分片调整 - 根据集群成员变化自动调整分片分配
  3. 集群事件处理 - 响应节点加入、离开和故障事件
  4. 远程节点代理 - 为远程节点创建本地代理

NodeDistributor​

NodeDistributor 是集群与分片的协调器，负责管理节点变更与一致性哈希环的同步：

// NodeDistributor 管理集群节点和一致性哈希环之间的映射关系
func (nd *NodeDistributor) HandleClusterEvent(event cache.ClusterEvent) {
    // ...
    switch event.Type {
    case cache.EventNodeJoin:
        // 节点加入时，创建远程缓存代理并添加到一致性哈希环
        
    case cache.EventNodeLeave, cache.EventNodeFailed:
        // 节点离开或故障时，从一致性哈希环中移除
        
    case cache.EventNodeRecovered:
        // 节点恢复时，重新添加到一致性哈希环
    }
    // ...
}
