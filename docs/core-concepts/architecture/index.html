System Architecture & Design Principles

本文档介绍FyerCache的系统架构设计和核心原则。

总体架构​

核心组件详解​

缓存接口层​

FyerCache的所有实现都基于统一的缓存接口：

type Cache interface {
    Get(ctx context.Context, key string) (any, error)
    Set(ctx context.Context, key string, val any, expiration time.Duration) error
    Del(ctx context.Context, key string) error
}

这个简洁的接口确保了不同缓存实现之间的一致性，无论是内存缓存还是分布式缓存，用户都能以相同的方式进行交互。

内存缓存层​

内存缓存层提供高性能的本地内存存储功能：

  1. MemoryCache：基础内存缓存实现，提供键值存储、过期机制和自动清理
  2. ShardedMap：内部采用分片设计，减少锁竞争以提高并发性能
  3. MemCacheMemory：带内存管理和LRU淘汰机制的内存缓存

内存缓存层通过选项模式提供高度定制能力，包括分片数量、淘汰回调、清理间隔等。

分片缓存层​

分片缓存层通过数据分片提高并发性能和可扩展性：

  1. 内部分片：使用ShardedMap将数据分散到多个分片，每个分片拥有独立的锁，以减少锁竞争
  2. 节点分片：NodeShardedCache使用一致性哈希将数据分布到多个缓存节点

一致性哈希​

一致性哈希是分布式缓存的核心机制，确保数据在节点变更时的稳定性：

  1. 虚拟节点：每个物理节点对应多个虚拟节点，以提高数据分布均匀性
  2. 节点权重：支持为不同节点设置不同权重，适应异构环境
  3. 副本因子：支持数据在多个节点上保存副本，提高可用性

一致性哈希实现保证了当节点加入或离开时，只有少部分数据需要重新分布，最小化数据迁移成本。

分布式层​

分布式层提供集群管理、数据同步和复制机制：

集群管理​

  1. Node：表示集群中的一个节点，管理集群成员和通信
  2. Membership：跟踪集群成员状态，处理节点加入/离开事件
  3. Gossip：基于Gossip协议传播集群成员信息，实现最终一致性
  4. Transport：节点间通信层，支持消息发送和处理

数据同步​

  1. SyncManager：管理节点间的数据同步，支持增量和全量同步
  2. Remote Cache：远程节点的缓存客户端，透明访问远程数据

复制机制​

  1. LeaderNode：主节点实现，处理写操作并复制到从节点
  2. FollowerNode：从节点实现，从主节点同步数据
  3. ReplicationLog：复制日志，记录和同步操作顺序

集群架构​

FyerCache支持两种主要的集群架构：

分片集群架构​

     客户端
       /|\
        |
  +-----+------+
  |            |
+--+--+      +--+--+
|Node1|      |Node2|
+--+--+      +--+--+
   |            |
+--+------------+--+
|  一致性哈希环     |
+------------------+

数据通过一致性哈希分布在所有节点上，每个节点负责一部分键空间，适合读写频繁的场景。

主从架构​

  读/写请求     只读请求
     |           /|\
     v            |
  +--+--+      +--+--+
  |主节点| ---> |从节点|
  +-----+      +-----+

主节点处理所有写操作并将更改复制到从节点，从节点只处理读操作，适合读多写少的场景。

监控层​

监控层提供运行时观测能力：

  1. Collector：收集缓存命中率、延迟、内存使用等指标
  2. Exporter：以Prometheus格式导出指标
  3. MetricsServer：暴露HTTP端点供监控系统抓取指标
  4. APIServer：提供管理API，如缓存统计和失效操作

核心设计原则​

FyerCache的设计遵循以下原则：

1. 分层抽象，接口清晰​

系统按功能划分为不同层次，每层都有明确的接口和责任：

  • 缓存层专注于高效数据存储和检索
  • 分片层负责数据分布和负载均衡
  • 分布式层处理集群通信和数据同步
  • 监控层提供可观测性

2. 组件正交，松耦合​

各组件之间通过接口交互，减少耦合：

  • 内存缓存与分片机制相互独立
  • 集群管理与缓存实现分离
  • 监控组件作为可选装饰器

这种设计使得系统各部分可以独立演化，方便测试和维护。

3. 选项模式，高度定制​

大量使用函数选项模式提供灵活配置：

cache := NewMemoryCache(
    WithShardCount(32),
    WithEvictionCallback(func(key string, value any) {
        // 自定义回调
    }),
    WithCleanupInterval(5*time.Minute),
)

这种方式既保持了API简洁，又提供了丰富的定制能力。

4. 容错设计，优雅降级​

系统设计具有容错性：

  • 节点故障自动检测和处理
  • 数据冗余存储，支持副本机制
  • 故障节点自动隔离与恢复
  • 网络分区后的一致性处理

数据流​

写入流程​

  1. 客户端调用 Set 方法设置缓存键值
  2. 如果是分片缓存，通过一致性哈希确定目标分片
  3. 如果是分布式缓存，确定目标节点并路由请求
  4. 在目标存储中写入数据
  5. 如果启用了主从复制，将操作记录到复制日志并同步到从节点
  6. 返回操作结果

读取流程​

  1. 客户端调用 Get 方法获取缓存键值
  2. 如果是分片缓存，通过一致性哈希确定目标分片
  3. 如果是分布式缓存，确定目标节点并路由请求
  4. 在目标存储中查询数据
  5. 如果数据不存在且配置了缓存策略（如 Cache-Aside），从数据源加载并写入缓存
  6. 返回数据或未找到错误

节点加入流程​

  1. 新节点启动并尝试加入集群
  2. 新节点连接一个种子节点并发送加入消息
  3. 种子节点将新节点信息通过Gossip协议传播到集群
  4. 所有节点更新一致性哈希环，重新分配部分键空间
  5. 数据按需从旧节点迁移到新节点

节点故障流程​

  1. 节点通过Gossip协议检测到某节点无响应
  2. 将该节点标记为可疑状态，并增加探测频率
  3. 如果节点持续无响应，标记为失败状态
  4. 更新一致性哈希环，将失败节点的键空间重新分配
  5. 如果启用了副本机制，从副本节点恢复数据

扩展性和可定制性​

FyerCache通过多种机制支持扩展：

  1. 接口驱动设计：所有核心功能通过接口定义，便于替换实现
  2. 中间件和装饰器：如监控装饰器、缓存一致性策略等
  3. 事件系统：节点事件和缓存事件允许构建自定义反应逻辑
  4. 插件化：支持自定义一致性哈希函数、淘汰策略等
