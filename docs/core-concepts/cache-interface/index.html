Cache Interface Design

核心 Cache 接口​

所有 FyerCache 实现的基础是简单而强大的 Cache 接口：

// Cache 基本缓存接口
type Cache interface {
    Get(ctx context.Context, key string) (any, error)
    Set(ctx context.Context, key string, val any, expiration time.Duration) error
    Del(ctx context.Context, key string) error
}

这个精简的接口定义了缓存系统最基本的三个操作：

  • Get - 从缓存中检索值
  • Set - 向缓存中存储值，可设置过期时间
  • Del - 从缓存中删除值

所有 Get、Set 和 Del 方法都接受一个 context.Context 参数，允许传递截止时间、取消信号和请求范围的值。

分片缓存接口​

对于扩展性更强的缓存实现，FyerCache 提供了分片缓存接口：

// ShardedCache 分片缓存接口
type ShardedCache interface {
    Cache  // 嵌入基本缓存接口

    // AddNode 添加缓存节点
    AddNode(nodeID string, cache Cache, weight int) error

    // RemoveNode 移除缓存节点
    RemoveNode(nodeID string) error

    // GetNodeIDs 获取所有节点ID
    GetNodeIDs() []string

    // GetNodeCount 获取节点数量
    GetNodeCount() int
}

ShardedCache 扩展了基本的 Cache 接口，增加了节点管理功能，适用于需要水平扩展的缓存场景。

一致性哈希接口​

一致性哈希是分布式缓存的核心机制，用于确定数据应该路由到哪个节点：

// Ring 一致性哈希环接口
type Ring interface {
    // Add 添加节点
    Add(node string, weight int)

    // Remove 移除节点
    Remove(node string)

    // Get 获取键对应的节点
    Get(key string) string

    // GetN 获取键对应的N个节点（用于备份）
    GetN(key string, count int) []string

    // GetNodes 获取所有节点
    GetNodes() []string
}

// HashFunc 定义哈希函数类型
type HashFunc func(data []byte) uint32

Ring 接口抽象了一致性哈希算法的核心操作，支持：

  • 添加和移除节点
  • 查找键对应的节点
  • 获取多个副本节点（用于数据冗余）
  • 自定义哈希函数

分布式缓存接口​

在分布式环境中，FyerCache 提供了更丰富的接口，支持跨节点操作：

// DistributedCache 是分布式缓存操作的接口
type DistributedCache interface {
    // 核心缓存操作
    Get(ctx context.Context, key string) (any, error)
    Set(ctx context.Context, key string, value any, expiration time.Duration) error
    Del(ctx context.Context, key string) error

    // 本地缓存操作，绕过分布式路由直接访问当前节点的数据
    GetLocal(ctx context.Context, key string) (any, error)
    SetLocal(ctx context.Context, key string, value any, expiration time.Duration) error
    DelLocal(ctx context.Context, key string) error

    // 节点管理
    NodeID() string
    NodeAddress() string
}

DistributedCache 接口拓展了核心功能，增加了本地操作方法，允许在必要时绕过分布式路由直接操作本地缓存。

集群节点接口​

集群功能由以下接口提供支持：

// ClusterNode 集群节点接口
type ClusterNode interface {
    // Join 加入集群
    Join(seedNodeAddr string) error

    // Leave 离开集群
    Leave() error

    // Start 启动节点服务
    Start() error

    // Stop 停止节点服务
    Stop() error

    // Members 获取集群成员列表
    Members() []NodeInfo

    // IsCoordinator 判断当前节点是否为协调节点
    IsCoordinator() bool

    // Events 返回事件通知通道
    Events() <-chan ClusterEvent
}

相关的 NodeInfo 结构定义了节点的元数据信息：

// NodeInfo 节点信息
type NodeInfo struct {
    ID       string
    Addr     string
    Status   NodeStatus
    LastSeen time.Time
    Metadata map[string]string
}

缓存一致性策略接口​

为了支持不同的缓存一致性模型，FyerCache 定义了一致性策略接口：

// ConsistencyStrategy 定义缓存一致性策略接口
type ConsistencyStrategy interface {
    // Get 从缓存获取数据，缓存不存在时从数据源获取
    Get(ctx context.Context, key string) (any, error)

    // Set 设置缓存数据并根据策略同步到数据源
    Set(ctx context.Context, key string, value any, expiration time.Duration) error

    // Del 删除缓存和数据源中的数据
    Del(ctx context.Context, key string) error

    // Invalidate 使缓存失效但不影响数据源
    Invalidate(ctx context.Context, key string) error
}

// DataSource 定义数据源接口
type DataSource interface {
    // Load 从数据源加载数据
    Load(ctx context.Context, key string) (any, error)

    // Store 将数据保存到数据源
    Store(ctx context.Context, key string, value any) error

    // Remove 从数据源删除数据
    Remove(ctx context.Context, key string) error
}

这些接口支持不同的缓存一致性策略实现，如 Cache-Aside、Write-Through 和基于消息队列的通知机制。

主从复制接口​

FyerCache 支持主从架构，通过以下接口定义复制功能：

// ReplicationLog 复制日志接口
type ReplicationLog interface {
    // Append 追加日志条目
    Append(entry *ReplicationEntry) error

    // Get 获取指定索引的日志
    Get(index uint64) (*ReplicationEntry, error)

    // GetFrom 获取从指定索引开始的所有日志
    GetFrom(startIndex uint64, maxCount int) ([]*ReplicationEntry, error)

    // GetLastIndex 获取最后一条日志的索引
    GetLastIndex() uint64

    // GetLastTerm 获取最后一条日志的任期
    GetLastTerm() uint64

    // Truncate 截断日志到指定索引
    Truncate(index uint64) error
}

// LeaderNode 领导者节点接口
type LeaderNode interface {
    // 节点生命周期方法
    Start() error
    Stop() error
    
    // 角色管理
    GetRole() ReplicationRole
    SetRole(role ReplicationRole)
    GetLeader() string
    
    // 从节点管理
    AddFollower(nodeID string, address string) error
    RemoveFollower(nodeID string) error
    GetFollowers() map[string]string
    
    // 复制操作
    ReplicateEntries(ctx context.Context) error
}

// FollowerNode 从节点接口
type FollowerNode interface {
    // 节点生命周期方法
    Start() error
    Stop() error
    
    // 角色管理
    GetRole() ReplicationRole
    SetRole(role ReplicationRole)
    GetLeader() string
    SetLeader(leaderID string, leaderAddr string)
    
    // 同步操作
    SyncFromLeader(ctx context.Context) error
    RequestFullSync(ctx context.Context) error
    
    // 心跳处理
    HandleHeartbeat(ctx context.Context, leaderID string, term uint64) error
}

通信传输接口​

节点间通信由以下接口提供：

// NodeTransport 节点通信传输层接口
type NodeTransport interface {
    // Send 发送消息到指定节点
    Send(nodeAddr string, msg *NodeMessage) (*NodeMessage, error)

    // Start 启动传输层服务
    Start() error

    // Stop 停止传输层服务
    Stop() error

    // Address 返回当前节点地址
    Address() string
}

接口层次关系​

FyerCache 的接口设计形成了清晰的层次结构：

  1. Core Layer - Cache 接口作为基础，定义核心缓存操作
  2. Distribution Layer - ShardedCache、Ring 接口支持数据分片和一致性哈希
  3. Cluster Layer - ClusterNode、NodeTransport 接口支持集群功能
  4. Consistency Layer - ConsistencyStrategy、DataSource 接口处理缓存一致性
  5. Replication Layer - ReplicationLog、LeaderNode、FollowerNode 接口处理数据复制

每一层都构建在前一层的基础上，通过接口组合和扩展，增加新的功能特性。

扩展与自定义​

FyerCache 的接口设计使其易于扩展，你可以：

  1. 实现自定义的 Cache 接口，创建新的缓存存储
  2. 实现自定义的 Ring 接口，提供替代的数据分布策略
  3. 实现自定义的 ConsistencyStrategy，满足特定的一致性需求
  4. 实现自定义的 NodeTransport，使用其他通信协议

所有这些接口都有默认实现，但系统的模块化设计允许根据需要替换任何组件。

示例：接口使用模式​

简单内存缓存：

// 使用基本缓存接口
cache := cache.NewMemoryCache()
cache.Set(ctx, "user:1001", userData, 5*time.Minute)
user, err := cache.Get(ctx, "user:1001")

分片缓存：

// 创建基于节点的分片缓存
shardedCache := cache.NewNodeShardedCache()

// 添加三个缓存节点
shardedCache.AddNode("node1", cache.NewMemoryCache(), 1)
shardedCache.AddNode("node2", cache.NewMemoryCache(), 1)
shardedCache.AddNode("node3", cache.NewMemoryCache(), 2) // 更高权重

// 使用方式与基本缓存相同
shardedCache.Set(ctx, "key1", "value1", time.Hour)

一致性策略：

// 创建 Cache-Aside 策略
cacheStrategy := consistency.NewCacheAsideStrategy(
    memCache, 
    dataSource,
    consistency.WithCacheAsideTTL(10*time.Minute),
)

// 使用策略获取数据
value, err := cacheStrategy.Get(ctx, "product:1001")

总结​

FyerCache 的接口设计通过分层和组合，在保持简洁性的同时提供了极大的灵活性和扩展性。这种设计使得 FyerCache 可以适应从单机应用到大规模分布式系统的各种缓存需求，同时保持一致的编程模型和操作语义。

接口的明确分离也使得系统更易于测试和维护，各组件可以独立开发和升级，而不会影响系统的其他部分。
