Failover Handling

FyerCache 实现了多层故障处理机制，确保在分布式环境中即使发生节点故障或网络分区，系统仍能保持高可用性并最终恢复一致性。

节点故障检测​

FyerCache 采用基于 Gossip 协议和主动探测相结合的方式检测节点故障：

// 节点健康检查循环
func (m *Membership) probeLoop() {
    ticker := time.NewTicker(m.probeInterval)
    defer ticker.Stop()

    for {
        select {
        case <-m.stopCh:
            return
        case <-ticker.C:
            m.probeNextNode()
        }
    }
}

可疑状态机制​

节点故障检测采用多阶段处理，避免因临时网络问题误判节点故障：

  1. 正常状态 (NodeStatusUp): 节点正常运行
  2. 可疑状态 (NodeStatusSuspect): 节点探测超时，进入可疑状态
  3. 失败状态 (NodeStatusDown): 持续多次探测失败，节点被标记为失败
// 将节点标记为可疑状态
func (m *Membership) MarkSuspect(nodeID string) {
    m.mu.Lock()
    defer m.mu.Unlock()
    
    if info, exists := m.members[nodeID]; exists {
        info.Status = cache.NodeStatusSuspect
        m.suspectTime[nodeID] = time.Now()
        // 发布状态变更事件
    }
}

// 将节点标记为失败状态
func (m *Membership) MarkFailed(nodeID string, reason string) {
    m.mu.Lock()
    defer m.mu.Unlock()
    
    if info, exists := m.members[nodeID]; exists {
        info.Status = cache.NodeStatusDown
        delete(m.suspectTime, nodeID)
        m.events.Publish(cluster.NewNodeFailedEvent(nodeID, reason))
    }
}

数据副本与容错​

FyerCache 通过副本机制实现数据冗余和故障容错：

副本因子​

副本因子控制每个键在集群中存储的副本数量：

// 设置备份因子选项
func WithNodeReplicaFactor(factor int) NodeShardedCacheOption {
    return func(nsc *NodeShardedCache) {
        if factor > 0 {
            nsc.replicaFactor = factor
        }
    }
}

当副本因子大于 1 时，每个键值对会在多个节点上存储副本：

// Set 将缓存项设置到正确的节点
func (nsc *NodeShardedCache) Set(ctx context.Context, key string, val any, expiration time.Duration) error {
    // 获取应存储键的所有节点（主节点+副本节点）
    nodeIDs := nsc.ring.GetN(key, nsc.replicaFactor)
    
    // 在所有节点上存储值
    // 即使备份节点失败，只要主节点成功，也认为操作成功
}

容错读取​

当主节点失败时，系统会从备份节点获取数据：

// Get 从正确的节点获取缓存项
func (nsc *NodeShardedCache) Get(ctx context.Context, key string) (any, error) {
    // 获取包含该键的所有节点
    nodeIDs := nsc.ring.GetN(key, nsc.replicaFactor)
    
    // 首先检查主节点
    value, err := nodes[0].Get(ctx, key)
    if err == nil {
        return value, nil
    }
    
    // 如果主节点失败，依次尝试从备份节点获取
    for i := 1; i < len(nodes) && i < len(nodeIDs); i++ {
        if val, err := nodes[i].Get(ctx, key); err == nil {
            return val, nil
        }
    }
    
    // 所有节点都失败时返回错误
    return nil, fmt.Errorf("all nodes failed")
}

节点加入与离开处理​

节点加入流程​

    // 触发与种子节点的同步
}

当新节点加入集群时：

  1. 新节点连接一个种子节点并发送加入消息
  2. 种子节点将新节点信息通过 Gossip 协议传播到集群
  3. 所有节点更新一致性哈希环
  4. 重新分配部分键空间，数据按需从旧节点迁移到新节点

节点离开流程​

// 节点优雅离开过程
func (n *Node) Leave() error {
    // 创建并向所有成员发送离开消息
    // 标记自己为离开状态
}

当节点离开集群时：

  1. 节点发送离开消息到其他节点
  2. 其他节点从一致性哈希环中移除该节点
  3. 被移除节点的键空间重新分配给剩余节点
  4. 如果启用了副本机制，数据从副本节点中恢复

主从架构的故障转移​

在主从架构中，FyerCache 实现了完整的故障转移机制：

心跳监控​

从节点通过心跳机制监控主节点的健康状态：

// 启动心跳监控定时器
func (f *FollowerNodeImpl) startHeartbeatMonitor() {
    f.heartbeatTimer = time.AfterFunc(f.heartbeatTimeout, func() {
        // 检查最后一次心跳时间
        // 如果超过超时时间未收到心跳，触发选举
        if f.onLeaderFail != nil {
            f.onLeaderFail()
        }
    })
}

领导者选举​

当主节点失败时，触发选举流程选出新的主节点：

// 设置选举回调函数
func (lfc *LeaderFollowerCluster) setElectionCallback() {
    if follower, ok := lfc.followerNode.(*replication.FollowerNodeImpl); ok {
        follower.SetElectionCallback(func() {
            // 发起选举
            lfc.handleFailover()
        })
    }
}

// 处理故障转移
func (lfc *LeaderFollowerCluster) handleFailover() {
    // 根据节点优先级、任期等因素决定是否成为新的领导者
    // 如果符合条件，提升为领导者
    lfc.PromoteToLeader()
}

远程缓存故障处理​

远程缓存客户端包含多层错误处理机制：

超时控制​

// 设置操作超时
func WithTimeout(timeout time.Duration) RemoteCacheOption {
    return func(rc *RemoteCache) {
        if timeout > 0 {
            rc.timeout = timeout
        }
    }
}

重试机制​

// 带重试的操作执行
func (rc *RemoteCache) doWithRetry(method, url string, body, result interface{}) error {
    // 初始尝试
    err := rc.client.Do(method, url, body, result)
    if err == nil {
        return nil
    }
    
    // 失败后进行重试
    for i := 0; i < rc.retries; i++ {
        time.Sleep(rc.interval)
        if err := rc.client.Do(method, url, body, result); err == nil {
            return nil
        }
    }
    
    return fmt.Errorf("all retry attempts failed: %w", err)
}

错误降级策略​

在实际应用中，FyerCache 支持以下错误降级策略：

缓存操作降级​

当缓存操作失败时，系统可以降级到直接访问数据源：

// 错误处理示例
func handleCacheError(ctx context.Context, key string, cache cache.Cache, dataSource DataSource) (any, error) {
    // 尝试从缓存获取
    value, err := cache.Get(ctx, key)
    if err == nil {
        return value, nil
    }
    
    // 缓存失败时直接查询数据源
    return dataSource.Load(ctx, key)
}

熔断器模式​

对于持续故障的远程节点，可以实现熔断器模式避免级联失败：

// 熔断器示例
func getWithCircuitBreaker(ctx context.Context, key string) (any, error) {
    // 检查熔断器状态
    if isCircuitOpen("node1") {
        return getFallbackValue(key)
    }
    
    // 尝试获取值
    value, err := remoteCache.Get(ctx, key)
    if err != nil {
        recordError("node1")
        
        // 如果错误次数超过阈值，触发熔断
        if shouldTripCircuitBreaker("node1") {
            triggerCircuitBreaker("node1")
            return getFallbackValue(key)
        }
        
        return nil, err
    }
    
    recordSuccess("node1")
    return value, nil
}
