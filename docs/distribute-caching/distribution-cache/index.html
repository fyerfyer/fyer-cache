Distributed Caching Overview

分布式缓存是 FyerCache 的核心功能之一，它通过将数据分散存储在多个节点上，提供水平扩展能力、高可用性和故障容错性。

分布式缓存架构​

FyerCache 的分布式缓存架构由以下主要组件构成：

核心组件​

  1. 分布式缓存接口 (DistributedCache)： 扩展基本缓存接口，增加分布式环境的感知能力。

    // DistributedCache 是分布式缓存操作的接口
    type DistributedCache interface {
        // 核心缓存操作
        Get(ctx context.Context, key string) (any, error)
        Set(ctx context.Context, key string, value any, expiration time.Duration) error
        Del(ctx context.Context, key string) error

        // 本地缓存操作，绕过分布式路由直接访问当前节点的数据
        GetLocal(ctx context.Context, key string) (any, error)
        SetLocal(ctx context.Context, key string, value any, expiration time.Duration) error
        DelLocal(ctx context.Context, key string) error

        // 节点管理
        NodeID() string
        NodeAddress() string
    }

  2. 集群实现：

    • ShardedClusterCache：基于分片的分布式缓存实现，使用一致性哈希分布数据
    • LeaderFollowerCluster：基于主从架构的分布式缓存实现，适合读多写少场景

  3. 集群管理：

    • Node：集群节点管理，处理节点的生命周期和通信
    • Membership：成员管理，跟踪集群中的节点状态
    • Gossip：基于 Gossip 协议的集群成员信息传播
    • Transport：节点间通信层，支持数据和控制信息交换

  4. 数据分布与同步：

    • ConsistentHash：一致性哈希算法，用于数据分片和负载均衡
    • NodeShardedCache：基于节点的分片缓存实现
    • SyncManager：节点间数据同步管理
    • Replication：主从数据复制系统

数据分布策略​

FyerCache 支持两种主要的数据分布策略：

1. 基于一致性哈希的分片​

在分片模式下，数据通过一致性哈希算法均匀分布在集群中的所有节点上：

// 创建分片缓存
shardedCache := cache.NewNodeShardedCache(
    cache.WithNodeHashReplicas(100),    // 设置虚拟节点数量
    cache.WithNodeReplicaFactor(2),     // 设置数据副本数
)

// 添加缓存节点
shardedCache.AddNode("node1", node1Cache, 1)  // 权重为1
shardedCache.AddNode("node2", node2Cache, 2)  // 权重为2

优势和特点：

  • 高扩展性：可以动态添加和移除节点，最小化数据迁移
  • 负载均衡：数据和请求均匀分布在所有节点上
  • 高可用性：通过副本机制实现数据冗余
  • 配置灵活：支持虚拟节点、节点权重和副本因子

2. 主从架构​

在主从模式下，写操作集中在主节点，而数据被复制到从节点以支持高可用性和读扩展：

// 创建主节点
leaderCluster, err := integration.NewLeaderFollowerCluster(
    localCache, "leader-1", "localhost:8000",
    []cluster.NodeOption{...},
    []integration.LeaderFollowerClusterOption{
        integration.WithInitialRole(replication.RoleLeader),
    },
)

// 创建从节点
followerCluster, err := integration.NewLeaderFollowerCluster(
    localCache, "follower-1", "localhost:8001",
    []cluster.NodeOption{...},
    []integration.LeaderFollowerClusterOption{
        integration.WithInitialRole(replication.RoleFollower),
    },
)

优势和特点：

  • 读写分离：写操作集中在主节点，读操作可以分散到多个从节点
  • 数据一致性：从主节点到从节点的数据复制确保一致性
  • 故障转移：主节点故障时可以选举新的主节点
  • 简化的数据分布：无需复杂的分片算法

集群管理​

FyerCache 使用多层机制管理分布式缓存集群：

成员管理​

Membership 组件负责跟踪集群中的节点状态：

  • 节点状态监控：节点可以处于 UP、SUSPECT、DOWN 和 LEFT 状态
  • 健康检查：定期发送探测消息验证节点健康状态
  • 故障检测：自动检测和标记失败节点

Gossip 协议​

FyerCache 使用 Gossip 协议在集群中传播成员信息，确保节点状态的最终一致性：

// Gossip协议配置
gossiper := NewGossiper(localID, membership, transport, config)
gossiper.Start()

Gossip 协议特性：

  • 去中心化：没有单点故障
  • 可扩展性：通信复杂度随节点数量增长缓慢
  • 弹性：能够处理网络分区和节点故障
  • 最终一致性：所有节点最终拥有相同的集群视图

事件系统​

FyerCache 的事件系统促进组件间的松耦合通信：

// 订阅集群事件
eventChannel := node.Events()
for event := range eventChannel {
    // 处理事件
    switch event.Type {
    case cache.EventNodeJoin:
        // 处理节点加入事件
    case cache.EventNodeLeave:
        // 处理节点离开事件
    case cache.EventNodeFailed:
        // 处理节点故障事件
    }
}

支持的事件类型：

  • 节点加入/离开事件
  • 节点故障/恢复事件
  • 数据同步完成事件
  • 领导者选举事件

数据复制与一致性​

复制机制​

FyerCache 在主从架构中实现了完整的复制系统：

  • 复制日志：记录所有写操作，提供幂等性和可重放性
  • 增量同步：高效传输变更数据
  • 全量同步：从主节点完整复制数据
// 复制日志条目
type ReplicationEntry struct {
    Index      uint64         // 日志索引
    Term       uint64         // 任期号
    Command    string         // 命令类型 (Set, Del)
    Key        string         // 缓存键
    Value      []byte         // 缓存值
    Expiration time.Duration  // 过期时间
    Timestamp  time.Time      // 操作时间戳
    ValueType  string         // 存储值类型
}

同步策略​

FyerCache 支持以下同步策略：

  1. 主从定期同步：从节点定期从主节点同步数据
  2. 操作触发同步：写操作触发同步到从节点
  3. 心跳同步：主节点周期性发送心跳和同步数据
  4. 异步复制：主节点写入成功后，异步复制到从节点

远程缓存客户端​

RemoteCache 提供了透明访问远程节点的能力：

// 创建远程缓存客户端
remoteCache := distributed.NewRemoteCache(
    "remote-node-1",
    "localhost:8080",
    distributed.WithTimeout(500*time.Millisecond),
    distributed.WithRetry(3, 200*time.Millisecond),
)

// 使用方式与本地缓存相同
remoteCache.Set(ctx, "user:1001", userData, time.Hour)
userData, err := remoteCache.Get(ctx, "user:1001")

特性：

  • 透明访问：使用与本地缓存相同的接口
  • 超时控制：可配置请求超时
  • 重试机制：自动重试失败的请求
  • 错误处理：网络故障和节点故障的优雅处理

故障处理​

FyerCache 实现了多层故障处理机制：

节点故障检测​

  • 心跳监控：定期心跳检测节点活性
  • 失败标记：节点故障时自动标记状态
  • 节点恢复：支持节点恢复后重新加入集群

数据可用性保障​

  • 数据副本：关键数据在多个节点上存储副本
  • 读取容错：主节点故障时，可从备份节点读取数据
  • 自动重平衡：节点变化时自动重新分配数据

主从故障转移​

  • 主节点监控：从节点监控主节点健康状态
  • 领导者选举：主节点失效时自动选举新的主节点
  • 状态恢复：新主节点接管服务并维持数据一致性
